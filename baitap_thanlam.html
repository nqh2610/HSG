Bài Tập Tham Lam

Bộ đề rèn luyện kỹ năng Tham Lam (Greedy) dành cho HSG Tin học

Bài 1: XẾP SỮA
XEPSUA.INP / XEPSUA.OUT

Nông trại có N con bò và M chai sữa. Con bò thứ i chỉ chấp nhận sữa có giá không quá pi đồng. Chai sữa thứ j có giá cj đồng. Mỗi con bò uống tối đa 1 chai, mỗi chai cho tối đa 1 con bò. Tìm số con bò được uống sữa nhiều nhất.

Dữ liệu vào (XEPSUA.INP):
  • Dòng 1: Hai số nguyên N, M (1 ≤ N, M ≤ 105)
  • Dòng 2: N số nguyên p1, p2, ..., pN (1 ≤ pi ≤ 109)
  • Dòng 3: M số nguyên c1, c2, ..., cM (1 ≤ cj ≤ 109)
Kết quả ra (XEPSUA.OUT):
  • Một số nguyên duy nhất — số con bò được uống sữa tối đa

XEPSUA.INP

4 5
3 5 2 4
4 1 6 2 3

XEPSUA.OUT

4
📝 Code C++ — E1. Sort + Matching 
// Bài 1: XEPSUA — Greedy sort + two-pointer matching
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("XEPSUA.INP");
    ofstream fout("XEPSUA.OUT");
    int n, m;
    fin >> n >> m;
    vector<int> p(n), c(m);
    for (int i = 0; i < n; i++) fin >> p[i];
    for (int i = 0; i < m; i++) fin >> c[i];
    sort(p.begin(), p.end());
    sort(c.begin(), c.end());
    int ans = 0, j = 0;
    for (int i = 0; i < n && j < m; i++) {
        if (c[j] <= p[i]) {
            ans++;
            j++;
        }
    }
    fout << ans;
    return 0;
}
💡 Ý tưởng: Dùng Sắp xếp (A9) + Two Pointers (A5).

🎯 Tham lam thế nào? Sắp xếp giá sữa tăng, giá chấp nhận của bò cũng tăng. Dùng 2 con trỏ i (sữa) và j (bò). So sánh: nếu sữa[i] ≤ bò[j] → ghép cặp, cả 2 tiến (i++, j++). Nếu sữa[i] > bò[j] → bò j quá keo kiệt, không mua nổi sữa rẻ nhất → bỏ bò j (j++).

🔎 Vì sao tham lam đúng? Hãy tưởng tượng: sữa đã sắp từ rẻ → đắt, bò cũng vậy. Con bò rẻ tiền nhất (chỉ trả ít) thì nên nhận sữa rẻ nhất vừa vặn — vì nếu cho nó sữa đắt hơn thì lãng phí sữa đắt đó (con bò khó tính hơn sau này sẽ không có sữa mà mua). Chứng minh hoán vị: Giả sử ghép bò rẻ với sữa đắt, bò đắt với sữa rẻ → bò đắt có thể không đủ tiền mua sữa đắt → mất 1 cặp → kết quả tệ hơn.

VD: Sữa = [2, 4, 6], Bò = [3, 5, 7]. Ghép: bò 3 mua sữa 2 ✓, bò 5 mua sữa 4 ✓, bò 7 mua sữa 6 ✓ → 3 cặp. Nếu bò 3 mua sữa 6 → bò 5+7 chỉ còn sữa 2,4 → 3 cặp (may mắn). Nhưng nếu sữa = [2, 4, 8], bò = [3, 5, 7]: bò 3 mua sữa 8? → bò 7 chỉ còn sữa 2,4 → 2 cặp, mất 1! O(N log N + M log M).
Bài 2: CHỌN ĐOẠN
CHONDOAN.INP / CHONDOAN.OUT

Có N hoạt động, hoạt động thứ i diễn ra từ thời điểm ai đến bi. Hai hoạt động chồng chéo nếu chúng có thời điểm chung (một hoạt động kết thúc đúng lúc hoạt động kia bắt đầu thì KHÔNG chồng chéo). Chọn nhiều hoạt động nhất sao cho không có hai hoạt động nào chồng chéo.

Dữ liệu vào (CHONDOAN.INP):
  • Dòng 1: Số nguyên N (1 ≤ N ≤ 105)
  • N dòng tiếp theo: Mỗi dòng hai số ai, bi (1 ≤ ai < bi ≤ 109)
Kết quả ra (CHONDOAN.OUT):
  • Một số nguyên duy nhất — số hoạt động tối đa chọn được

📝 Code C++ — E2. Activity Selection 
// Bài 2: CHONDOAN — Greedy sort by end time
#include <bits/stdc++.h>
using namespace std;

bool cmp(pair<int,int> a, pair<int,int> b) {
    return a.second < b.second;
}

int main() {
    ifstream fin("CHONDOAN.INP");
    ofstream fout("CHONDOAN.OUT");
    int n;
    fin >> n;
    vector<pair<int,int>> a(n);
    for (int i = 0; i < n; i++) fin >> a[i].first >> a[i].second;
    sort(a.begin(), a.end(), cmp);
    int ans = 0, last = 0;
    for (int i = 0; i < n; i++) {
        if (a[i].first >= last) {
            ans++;
            last = a[i].second;
        }
    }
    fout << ans;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam — Activity Selection (B2).

🎯 Tham lam thế nào? Sắp xếp hoạt động theo giờ KẾT THÚC tăng. Duyệt lần lượt: nếu hoạt động này bắt đầu ≥ thời điểm kết thúc hoạt động đã chọn gần nhất → chọn nó (cập nhật mốc kết thúc). Ngược lại → bỏ qua (bị trùng giờ).

🔎 Vì sao tham lam đúng? Vì sao sắp theo kết thúc chứ không phải bắt đầu? Hãy tưởng tượng bạn đang xếp lịch phòng họp. Bạn muốn chọn nhiều cuộc họp nhất. Nếu chọn cuộc kết thúc sớm, phòng họp được giải phóng sớm → còn nhiều thời gian nhất cho các cuộc sau. Nếu bạn chọn cuộc kết thúc muộn hơn, nó "chiếm" phòng lâu hơn → mất cơ hội cho các cuộc sau. Chứng minh hoán vị: Nếu phương án tối ưu chọn hoạt động X (kết thúc muộn) thay vì Y (kết thúc sớm), thì thay X bằng Y: Y kết thúc sớm hơn → tất cả hoạt động sau X vẫn hợp lệ → số lượng ≥ phương án có X → chọn Y không bao giờ tệ hơn.

VD: Hoạt động: A(1-3), B(2-5), C(4-6), D(6-8). Sắp theo kết thúc: A(1-3), B(2-5), C(4-6), D(6-8). Chọn A → mốc=3. B bắt đầu 2 < 3 → bỏ. C bắt đầu 4 ≥ 3 → chọn → mốc=6. D bắt đầu 6 ≥ 6 → chọn → 3 hoạt động. Nếu chọn B trước: B(2-5) → mốc=5, C bỏ, D chọn → chỉ 2! O(N log N).
Bài 3: GHÉP SỐ
GHEPSO.INP / GHEPSO.OUT

Cho dãy N số nguyên dương (N chẵn). Ghép chúng thành N/2 cặp. Tổng của mỗi cặp là tổng hai số trong cặp. Hãy tìm cách ghép sao cho tổng lớn nhất trong các cặp là nhỏ nhất có thể.

Dữ liệu vào (GHEPSO.INP):
  • Dòng 1: Số nguyên N (2 ≤ N ≤ 105, N chẵn)
  • Dòng 2: N số nguyên a1, ..., aN (1 ≤ ai ≤ 106)
Kết quả ra (GHEPSO.OUT):
  • Một số nguyên duy nhất — giá trị tổng lớn nhất nhỏ nhất

GHEPSO.INP

6
1 3 5 2 8 4

GHEPSO.OUT

9
📝 Code C++ — E3. Pair Balance 
// Bài 3: GHEPSO — Sắp xếp, ghép nhỏ nhất với lớn nhất
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("GHEPSO.INP");
    ofstream fout("GHEPSO.OUT");
    int n;
    fin >> n;
    vector<int> a(n);
    for (int i = 0; i < n; i++) fin >> a[i];
    sort(a.begin(), a.end());
    int ans = 0;
    for (int i = 0; i < n / 2; i++)
        ans = max(ans, a[i] + a[n - 1 - i]);
    fout << ans;
    return 0;
}
💡 Ý tưởng: Dùng Sắp xếp (A9) + Tham lam (B2).

🎯 Tham lam thế nào? Sắp xếp mảng tăng dần. Ghép đầu + cuối, đầu+1 + cuối−1, ... (tức a[0]+a[n−1], a[1]+a[n−2], ...). Đáp án = max trong các tổng cặp.

🔎 Vì sao ghép nhỏ nhất + lớn nhất là tối ưu? Hãy nghĩ thế này: số lớn nhất (a[n−1]) BẮT BUỘC phải nằm trong 1 cặp nào đó. Nếu ghép nó với số lớn (ví dụ a[n−2]), tổng cặp = rất lớn → max bị đẩy lên cao. Nếu ghép nó với số nhỏ nhất (a[0]), tổng cặp = a[0]+a[n−1] — đây là cách "san bằng", giữ max thấp nhất có thể. Chứng minh hoán vị: Giả sử ghép a[0] với a[k] (k ≠ n−1) và a[j] với a[n−1] (j ≠ 0, j > 0). Ta có: a[j]+a[n−1] ≥ a[0]+a[n−1] (vì a[j] ≥ a[0]) → max cặp ≥ a[j]+a[n−1] ≥ a[0]+a[n−1]. Đổi lại ghép a[0]+a[n−1] chỉ có thể giảm hoặc giữ max → tốt hơn hoặc bằng.

VD: [1,2,3,4] → ghép (1,4)=5, (2,3)=5 → max = 5. Ghép khác (1,2)=3, (3,4)=7 → max = 7 — tệ hơn! Ghép (1,3)=4, (2,4)=6 → max = 6 — vẫn tệ hơn! Chỉ có cách min+max mới cho max nhỏ nhất. O(N log N).
Bài 4: THUYỀN BUỒM
THUYENBUOM.INP / THUYENBUOM.OUT

Có N người muốn qua sông. Mỗi thuyền chở tối đa 2 người và tổng trọng lượng không vượt quá C. Mỗi người có trọng lượng wi (đảm bảo wi ≤ C). Tìm số thuyền ít nhất để chở hết tất cả mọi người.

Dữ liệu vào (THUYENBUOM.INP):
  • Dòng 1: Hai số nguyên N, C (1 ≤ N ≤ 105; 1 ≤ C ≤ 109)
  • Dòng 2: N số nguyên w1, ..., wN (1 ≤ wi ≤ C)
Kết quả ra (THUYENBUOM.OUT):
  • Một số nguyên duy nhất — số thuyền ít nhất

THUYENBUOM.INP

5 100
40 60 80 50 30

THUYENBUOM.OUT

3
📝 Code C++ — E4. Two-Pointer 
// Bài 4: THUYENBUOM — Greedy two-pointer
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("THUYENBUOM.INP");
    ofstream fout("THUYENBUOM.OUT");
    int n, C;
    fin >> n >> C;
    vector<int> w(n);
    for (int i = 0; i < n; i++) fin >> w[i];
    sort(w.begin(), w.end());
    int left = 0, right = n - 1, boats = 0;
    while (left <= right) {
        if (left == right) {
            boats++;
            break;
        }
        if (w[left] + w[right] <= C) left++;
        right--;
        boats++;
    }
    fout << boats;
    return 0;
}
💡 Ý tưởng: Dùng Two Pointers ngược chiều (A5) + Tham lam (B2).

🎯 Tham lam thế nào? Sắp xếp trọng lượng tăng dần. Đặt left = 0 (nhẹ nhất), right = n−1 (nặng nhất). Lặp: nếu a[left] + a[right] ≤ W (sức chứa thuyền) → 2 người lên chung 1 thuyền, left++, right−−. Nếu không vừa → người nặng (right) ĐI MỘT MÌNH, right−−. Dù cách nào cũng cần 1 thuyền → boats++.

🔎 Vì sao tham lam đúng? Tại sao ghép nhẹ nhất + nặng nhất? Người nặng nhất (right) BẮT BUỘC phải lên thuyền. Câu hỏi là: ai đi cùng? (a) Nếu ngay cả người nhẹ nhất cũng không vừa → không ai đi cùng được → right đi 1 mình, không mất gì. (b) Nếu người nhẹ nhất vừa → ghép 2 người, tiết kiệm 1 thuyền. Tại sao không để người nhẹ ghép với ai khác "tốt hơn"? Không có "tốt hơn" — mỗi thuyền tối đa 2 người, ghép được là tốt rồi, ghép bất kỳ đều tiết kiệm đúng 1 thuyền. Vậy ghép với người nặng nhất = dùng người nhẹ ở chỗ "khó nhất" (người nặng khó tìm bạn). Chứng minh: Giả sử PA tối ưu ghép nhẹ nhất (a[0]) với người khác (a[k], k ≠ n−1). Đổi: cho a[0] ghép a[n−1], a[k] ghép bạn cũ của a[n−1] (hoặc đi 1 mình). Vì a[0] ≤ a[k], nếu a[k]+a[n−1] ≤ W thì a[0]+a[n−1] ≤ W (vẫn vừa). Số thuyền không tăng → ít nhất bằng PA tối ưu.

VD: W = 5, trọng lượng [1, 2, 3, 4]. left=0, right=3: 1+4=5 ≤ 5 → ghép → 1 thuyền. left=1, right=2: 2+3=5 ≤ 5 → ghép → 1 thuyền. Tổng = 2 thuyền (tối ưu, vì 4 người tối thiểu 2 thuyền). O(N log N).
Bài 5: SÂN BAY
SANBAY.INP / SANBAY.OUT

Sân bay có N chuyến bay, chuyến thứ i đến lúc ai và khởi hành lúc di. Cổng được sử dụng trong khoảng [ai, di]. Một chuyến bay rời cổng lúc di thì cổng có thể nhận chuyến mới đến cũng lúc di. Tìm số cổng tối thiểu cần thiết để phục vụ tất cả chuyến bay.

// Bài 5: SANBAY — Sweep line (quét sự kiện)
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("SANBAY.INP");
    ofstream fout("SANBAY.OUT");
    int n;
    fin >> n;
    // Sự kiện: (thời gian, loại). Đến = +1, Đi = -1
    vector<pair<int,int>> ev;
    for (int i = 0; i < n; i++) {
        int a, d;
        fin >> a >> d;
        ev.push_back(make_pair(a, 1));
        ev.push_back(make_pair(d, -1));
    }
    // Sort: cùng thời điểm, xử lý đi (-1) trước đến (+1)
    sort(ev.begin(), ev.end());
    int cur = 0, ans = 0;
    for (int i = 0; i < (int)ev.size(); i++) {
        cur += ev[i].second;
        ans = max(ans, cur);
    }
    fout << ans;
    return 0;
}
💡 Ý tưởng: Dùng Sweep Line / Mảng hiệu (A3).

🎯 Kỹ thuật Sweep Line thế nào? Mỗi chuyến bay tạo 2 sự kiện: (thời_đến, +1) và (thời_đi, −1). Gom tất cả sự kiện vào 1 mảng, sắp xếp theo thời gian. Quét trái→phải, cộng dồn biến đếm: gặp +1 thì đếm++, gặp −1 thì đếm−−. Max của biến đếm = số cổng cần.

🔎 Vì sao đúng? Tại mỗi thời điểm, biến đếm = số máy bay đang ĐỒNG THỜI ở sân bay. Mỗi máy bay cần 1 cổng riêng → số cổng tối thiểu = max số máy bay cùng lúc. Tại sao cùng thời điểm thì xử lý −1 trước +1? Vì máy bay đi trả cổng trước, rồi máy bay đến mới dùng cổng — cùng thời điểm thì không cần thêm cổng, giống xe bus trả khách xong mới đón khách mới. Trick: dùng pair(thời_gian, loại) — loại −1 < +1 nên sort tự xử lý đúng thứ tự.

VD: 3 chuyến: A(1-4), B(2-6), C(5-7). Sự kiện: (1,+1), (2,+1), (4,−1), (5,+1), (6,−1), (7,−1). Quét: t=1→1, t=2→2, t=4→1, t=5→2, t=6→1, t=7→0. Max = 2 → cần 2 cổng. Đúng vì A và B trùng nhau (1-4 ∩ 2-6). O(N log N).
Bài 6: BA LÔ PHÂN LƯỢNG
BALOPL.INP / BALOPL.OUT

Có N vật phẩm, vật thứ i có khối lượng wi và giá trị vi. Ba lô chứa tối đa W đơn vị khối lượng. Khác với bài toán ba lô 0/1, ở đây có thể lấy một phần (phân số) của mỗi vật. Tìm tổng giá trị lớn nhất có thể mang theo.

Dữ liệu vào (BALOPL.INP):
  • Dòng 1: Hai số nguyên N, W (1 ≤ N ≤ 105; 1 ≤ W ≤ 109)
  • N dòng tiếp theo: Mỗi dòng hai số wi, vi (1 ≤ wi, vi ≤ 106)
Kết quả ra (BALOPL.OUT):
  • Một số thực — giá trị lớn nhất (in chính xác 2 chữ số thập phân)

📝 Code C++ — E6. Fractional Knapsack 
// Bài 6: BALOPL — Greedy sort by value/weight ratio
#include <bits/stdc++.h>
using namespace std;

struct Item { int w, v; };

bool cmp(Item a, Item b) {
    return (long long)a.v * b.w > (long long)b.v * a.w;
}

int main() {
    ifstream fin("BALOPL.INP");
    ofstream fout("BALOPL.OUT");
    int n, W;
    fin >> n >> W;
    vector<Item> items(n);
    for (int i = 0; i < n; i++) fin >> items[i].w >> items[i].v;
    sort(items.begin(), items.end(), cmp);
    double ans = 0;
    int rem = W;
    for (int i = 0; i < n && rem > 0; i++) {
        if (items[i].w <= rem) {
            ans += items[i].v;
            rem -= items[i].w;
        } else {
            ans += (double)items[i].v * rem / items[i].w;
            rem = 0;
        }
    }
    fout << fixed << setprecision(2) << ans;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam — Fractional Knapsack (B2).

🎯 Tham lam thế nào? Tính đơn giá = giá_trị / khối_lượng cho mỗi vật. Sắp xếp giảm theo đơn giá. Duyệt từ vật đắt nhất: nếu balo còn đủ chỗ → lấy trọn. Nếu không đủ → lấy phần (fraction) vừa đủ chỗ còn lại. Trick code: So sánh đơn giá bằng nhân chéo v1×w2 > v2×w1 (tránh chia số thực gây sai số).

🔎 Vì sao tham lam đúng? Đây là điểm khác biệt quan trọng với Balo 0/1! Balo 0/1 (lấy nguyên hoặc không) thì tham lam SAI — vì có thể bỏ vật đắt nặng để lấy 2 vật rẻ nhẹ tổng tốt hơn. Nhưng Fractional (được cắt lẻ): 1 kg vật đắt LUÔN tốt hơn 1 kg vật rẻ, vì đơn giá cao hơn. Không tồn tại tình huống lấy vật rẻ lại có lợi hơn — bạn đã lấy vật đắt hết rồi, phần còn thừa mới lấy vật rẻ hơn. // Bài 7: NOITEP — Min-heap, luôn nối 2 tệp nhỏ nhất
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("NOITEP.INP");
    ofstream fout("NOITEP.OUT");
    int n;
    fin >> n;
    priority_queue<long long, vector<long long>, greater<long long>> pq;
    for (int i = 0; i < n; i++) {
        long long x;
        fin >> x;
        pq.push(x);
    }
    long long ans = 0;
    while (pq.size() > 1) {
        long long a = pq.top(); pq.pop();
        long long b = pq.top(); pq.pop();
        ans += a + b;
        pq.push(a + b);
    }
    fout << ans;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam (B2) + Priority Queue / Heap (C3).

🎯 Tham lam thế nào? Đẩy tất cả tệp vào min-heap. Lặp: rút 2 tệp nhỏ nhất ra, nối chúng (chi phí = tổng 2 tệp), đẩy tệp mới vào heap. Tiếp tục cho đến khi heap còn 1 tệp. Tổng chi phí = đáp án.

🔎 Vì sao luôn nối 2 tệp nhỏ nhất? Hãy hình dung cây Huffman: mỗi lần nối = tạo 1 nút cha. Tệp ban đầu = lá. Chi phí = tổng (kích thước lá × độ sâu lá). Tệp nhỏ nên ở SÂU (nối sớm, bị cộng nhiều lần nhưng nhỏ nên ít tốn), tệp lớn nên ở NÔNG (nối muộn, ít bị cộng lại). Giống cách nén Huffman: ký tự ít xuất hiện → mã dài, ký tự nhiều → mã ngắn. Chứng minh: Giả sử nối 2 tệp KHÔNG nhỏ nhất (a, b) mà a > c (c là tệp nhỏ nhất bị bỏ qua). Đổi: nối c thay a, chi phí bước này = c+b < a+b. Các bước sau: tệp mới nhỏ hơn → tổng chi phí giảm → PA ban đầu không tối ưu.

// Bài 8: NHIENLIEU — Đi xa nhất có thể, đổ khi cần
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("NHIENLIEU.INP");
    ofstream fout("NHIENLIEU.OUT");
    int L, C, n;
    fin >> L >> C >> n;
    vector<int> d(n + 2);
    d[0] = 0;
    for (int i = 1; i <= n; i++) fin >> d[i];
    d[n + 1] = L;
    // Kiểm tra khả thi: mọi khoảng cách liên tiếp ≤ C
    for (int i = 1; i <= n + 1; i++) {
        if (d[i] - d[i - 1] > C) {
            fout << -1;
            return 0;
        }
    }
    int fuel = C, stops = 0;
    for (int i = 1; i <= n + 1; i++) {
        fuel -= (d[i] - d[i - 1]);
        // Tại trạm i: nếu không đủ xăng đến trạm tiếp, đổ đầy
        if (i <= n && fuel < d[i + 1] - d[i]) {
            fuel = C;
            stops++;
        }
    }
    fout << stops;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam (B2).

🎯 Tham lam thế nào? Thêm điểm xuất phát (0) và đích (L) vào danh sách trạm. Duyệt trạm từ trái sang phải. Tại mỗi trạm: tính khoảng cách đến trạm tiếp theo. Nếu xăng còn đủ → KHÔNG đổ, đi tiếp. Nếu không đủ → đổ thêm VỪA ĐỦ để đến trạm kế. Nếu khoảng cách > C (bình đầy cũng không tới) → in −1.

🔎 Vì sao "đổ ít nhất, chỉ đổ khi cần" là tối ưu? Bài này tối thiểu SỐ LẦN DỪNG (không phải lượng xăng). Mỗi lần dừng đổ xăng = tốn 1 bước. Nếu bạn xăng còn đủ đến trạm kế mà vẫn dừng → lãng phí 1 lần dừng. Vậy chỉ dừng khi BẮT BUỘC (xăng không đủ). Chứng minh: Giả sử PA tối ưu dừng ở trạm i (nhưng xăng đủ để qua trạm i). Bỏ lần dừng đó → vẫn đến được trạm kế → số lần dừng giảm 1 → PA ban đầu không tối ưu, mâu thuẫn.

Đi 2→ còn 0. Đến trạm 5: cần 3, còn 0 → đổ 3 (stops=2), xăng=3. Đi 3→ còn 0. Đến trạm 8: cần 2, còn 0 → đổ 2 (stops=3). Đến đích. Tổng = 3 lần dừng. O(N).

// Bài 9: NHAYBAC — Greedy BFS-like (Jump Game II)
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("NHAYBAC.INP");
    ofstream fout("NHAYBAC.OUT");
    int n;
    fin >> n;
    vector<int> a(n);
    for (int i = 0; i < n; i++) fin >> a[i];
    if (n == 1) { fout << 0; return 0; }
    // curEnd: biên phải vùng đang xét, farthest: xa nhất có thể tới
    int jumps = 0, curEnd = 0, farthest = 0;
    for (int i = 0; i < n - 1; i++) {
        farthest = max(farthest, i + a[i]);
        if (i == curEnd) {
            if (farthest == curEnd) {
                fout << -1;
                return 0;
            }
            jumps++;
            curEnd = farthest;
            if (curEnd >= n - 1) break;
        }
    }
    fout << jumps;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam (B2).

🎯 Tham lam thế nào? Hình dung như BFS theo tầng. Mỗi "bước nhảy" = 1 tầng. Giữ 2 biến: curEnd = biên phải của tầng hiện tại (từ đâu đến đây bằng số bước hiện tại), farthest = xa nhất có thể nhảy được từ các ô trong tầng này. Duyệt i từ 0: cập nhật farthest = max(farthest, i+a[i]). Khi i == curEnd: hết 1 tầng → jumps++, curEnd = farthest. Nếu farthest ≥ n−1 → đến được.

🔎 Vì sao tham lam đúng? Tại sao không cần thử tất cả đường? Ở mỗi tầng, bạn có nhiều ô có thể nhảy đến. Bạn chọn ô nào không quan trọng — điều quan trọng là từ TẤT CẢ các ô trong tầng, ô xa nhất bạn có thể vươn tới là farthest. Vì bất kỳ ô nào tầng sau nằm trong [đầu tầng, farthest] đều đạt được bằng đúng 1 bước nhảy → chọn farthest làm biên tầng mới = phủ được nhiều nhất → ít tầng (bước) nhất. Chứng minh: Tầng k của tham lam bao phủ [0, curEnd_k]. Bất kỳ PA nào khác với k bước, phải chọn 1 ô mỗi tầng — farthest của nó ≤ farthest của tham lam (vì tham lam xét TẤT CẢ ô) → curEnd PA ≤ curEnd tham lam → tham lam không bao giờ cần nhiều bước hơn.

VD: a = [2, 3, 1, 1, 4]. i=0: far=max(0,0+2)=2. i=1: far=max(2,1+3)=4. i=curEnd=0→? Không, curEnd ban đầu=0. À đúng, khi i==0==curEnd: jumps=1, curEnd=2. i=1: far=4. i=2==curEnd: jumps=2, curEnd=4 ≥ 4 (n−1) → đến! Đáp án: 2 bước. O(N).

// Bài 10: PHUDOAN — Sort by left, greedy extend
#include <bits/stdc++.h>
using namespace std;

bool cmp(pair<int,int> a, pair<int,int> b) {
    if (a.first != b.first) return a.first < b.first;
    return a.second > b.second;
}

int main() {
    ifstream fin("PHUDOAN.INP");
    ofstream fout("PHUDOAN.OUT");
    int A, B, n;
    fin >> A >> B >> n;
    vector<pair<int,int>> seg(n);
    for (int i = 0; i < n; i++) fin >> seg[i].first >> seg[i].second;
    sort(seg.begin(), seg.end(), cmp);
    int ans = 0, cur = A, idx = 0;
    while (cur < B) {
        int best = cur;
        while (idx < n && seg[idx].first <= cur) {
            best = max(best, seg[idx].second);
            idx++;
        }
        if (best == cur) {
            fout << -1;
            return 0;
        }
        cur = best;
        ans++;
    }
    fout << ans;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam (B2) + Sắp xếp (A9).

🎯 Tham lam thế nào? Sắp đoạn theo đầu trái tăng. Giữ biến covered = điểm đã phủ được (ban đầu = 0). Lặp: trong tất cả đoạn có đầu trái ≤ covered (có thể nối tiếp đoạn đã phủ), chọn đoạn có đầu phải XA NHẤT. Cập nhật covered = đầu phải đó. Lặp đến khi covered ≥ L (phủ hết) hoặc không tiến được (không đoạn nào bắt đầu ≤ covered).

🔎 Vì sao chọn đoạn vươn xa nhất là tối ưu? Tại mỗi bước, bạn CHẮC CHẮN phải chọn 1 đoạn nào đó (không chọn thì không phủ được). Trong số những đoạn có thể chọn (đầu trái ≤ covered), đoạn vươn xa nhất cho bạn nhiều "tiền độ" nhất — covered tăng nhiều nhất → cần ít đoạn nhất. Chọn đoạn ngắn hơn chỉ làm covered tăng chậm hơn → cần thêm đoạn. Chứng minh hoán vị: Giả sử PA tối ưu chọn đoạn A (phải ngắn hơn) thay vì B (phải xa hơn). Thay A bằng B: covered sau bước này ≥ trước → các bước sau vẫn hợp lệ (vì có nhiều đoạn hơn để chọn) → số đoạn ≤ PA ban đầu.

// Bài 11: PHATKEO — Duyệt 2 lượt trái→phải, phải→trái
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("PHATKEO.INP");
    ofstream fout("PHATKEO.OUT");
    int n;
    fin >> n;
    vector<int> r(n);
    for (int i = 0; i < n; i++) fin >> r[i];
    vector<int> candy(n, 1);
    // Lượt 1: trái → phải
    for (int i = 1; i < n; i++)
        if (r[i] > r[i - 1]) candy[i] = candy[i - 1] + 1;
    // Lượt 2: phải → trái
    for (int i = n - 2; i >= 0; i--)
        if (r[i] > r[i + 1]) candy[i] = max(candy[i], candy[i + 1] + 1);
    long long ans = 0;
    for (int i = 0; i < n; i++) ans += candy[i];
    fout << ans;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam 2 lượt (B2).

🎯 Tham lam thế nào? Khởi tạo candy[i] = 1 (ất nhất 1 kẹo mỗi bạn). Lượt 1 (trái→phải): nếu điểm[i] > điểm[i−1] thì candy[i] = candy[i−1]+1. Lượt 2 (phải→trái): nếu điểm[i] > điểm[i+1] thì candy[i] = max(candy[i], candy[i+1]+1). Tổng candy = đáp án.

🔎 Vì sao cần 2 lượt? Vì sao không 1 lượt? Mỗi bạn có 2 ràng buộc: phải nhiều hơn hàng xóm BÊN TRÁI (nếu giỏi hơn) VÀ hàng xóm BÊN PHẢI (nếu giỏi hơn). 1 lượt chỉ giải quyết được 1 hướng. Lượt 1 (→) đảm bảo ràng buộc trái. Lượt 2 (←) đảm bảo ràng buộc phải. Lấy max để thỏa MẤN CẢ HAI hướng với số kẹo nhỏ nhất. Tại sao max chứ không phải cộng? Vì candy[i] chỉ cần đủ lớn hơn CẢ 2 hàng xóm, max = giá trị nhỏ nhất thỏa cả 2 — cộng thì lãng phí.

VD: Điểm [1, 3, 5, 2, 1]. Lượt 1 (→): candy = [1, 2, 3, 1, 1] (5>3>1 tăng dần, 2<5 nên reset, 1<2 reset). Lượt 2 (←): i=3: điểm 2 > 1 → candy[3] = max(1, 1+1) = 2. i=2: 5 > 2 → candy[2] = max(3, 2+1) = 3. i=1: 3 > 5? Không → giữ. i=0: 1 > 3? Không → giữ. Kết quả: [1, 2, 3, 2, 1]. Tổng = 9. Kiểm tra: bạn 5 điểm (được 3 kẹo) > bạn 3 (2 kẹo) ✓ và > bạn 2 (2 kẹo) ✓. O(N).

int main() {
    ifstream fin("SOTOTHAT.INP");
    ofstream fout("SOTOTHAT.OUT");
    int n;
    fin >> n;
    vector<string> a(n);
    for (int i = 0; i < n; i++) {
        int x;
        fin >> x;
        a[i] = to_string(x);
    }
    sort(a.begin(), a.end(), cmp);
    // Trường hợp toàn số 0
    if (a[0] == "0") {
        fout << "0";
        return 0;
    }
    string result;
    for (int i = 0; i < n; i++) result += a[i];

💡 Ý tưởng: Dùng Sắp xếp tùy chỉnh (A9).

🎯 Tham lam thế nào? Chuyển mỗi số thành chuỗi. Sắp xếp với hàm so sánh: a đứng trước b nếu ghép a+b > b+a (so sánh chuỗi). Nối tất cả lại = đáp án. Lưu ý: nếu số lớn nhất là "0" thì tất cả = 0, in "0".

🔎 Vì sao so sánh ab với ba là đúng? Hãy nghĩ về 2 số liên kề trong kết quả: ...XY... vs ...YX... Phần còn lại giống nhau, chỉ khác XY hoặc YX. Nếu XY > YX thì X đứng trước là tốt hơn. Vậy với mỗi cặp liên kề, đặt đúng thứ tự → TOÀN CỤC tối ưu. Chứng minh: Nếu có cặp liên kề (X, Y) mà YX > XY (sai thứ tự), đổi thành (Y, X) → kết quả lớn hơn. Lặp cho đến không còn cặp sai → đúng là sắp xếp theo hàm so sánh đó. (Đây giống Bubble Sort: đổi cặp sai thì tăng, không còn cặp sai thì tối ưu.)

VD: Số: [3, 30, 34, 5, 9]. So sánh: 9 và 5: "95" > "59" → 9 trước 5. 9 và 34: "934" > "349" → 9 trước. 5 và 34: "534" > "345" → 5 trước. Thứ tự: 9, 5, 34, 3, 30 → "9534330". Kiểm: "3" và "30": "330" > "303" → đúng. O(N log N).

📝 Code C++ — E13. Stack Greedy 
// Bài 13: XAUMIN — Ngăn xếp đơn điệu tăng
#include <bits/stdc++.h>
using namespace std;

int main() {
    ifstream fin("XAUMIN.INP");
    ofstream fout("XAUMIN.OUT");
    string s;
    int k;
    fin >> s >> k;
    string result;
    for (int i = 0; i < (int)s.size(); i++) {
        // Bỏ đỉnh stack nếu nó lớn hơn ký tự hiện tại và còn được xóa
        while (k > 0 && !result.empty() && result.back() > s[i]) {
            result.pop_back();
            k--;
        }
        result.push_back(s[i]);
    }
    // Nếu K còn dư, xóa từ cuối
    while (k > 0) { result.pop_back(); k--; }
    fout << result;
    return 0;
}
💡 Ý tưởng: Dùng Monotonic Stack (A7) + Tham lam (B2).

🎯 Tham lam thế nào? Muốn số nhỏ nhất sau khi xóa K chữ số. Nguyên tắc: chữ số bên TRÁI có trọng số lớn hơn (hàng nghìn > hàng trăm > ...). Vậy phải ưu tiên xóa chữ số LỚN Ở VỊ TRÍ CAO. Dùng stack tăng dần: duyệt từ trái→phải, với mỗi chữ số c: khi đỉnh stack > c và còn quota xóa (K>0) → pop stack (xóa chữ số lớn ở vị trí cao). Rồi push c vào. Cuối cùng nếu K còn dư → xóa từ cuối stack (vì stack đã tăng, cuối là lớn nhất).

🔎 Vì sao tham lam đúng? Hãy nghĩ thế này: nếu có 2 chữ số liên tiếp d[i] > d[i+1], xóa d[i] chắc chắn tốt hơn xóa d[i+1] — vì d[i] ở vị trí cao hơn (nhiều đơn vị hơn) và lại lớn hơn. Xóa nó = thay thế bằng d[i+1] nhỏ hơn ở vị trí cao = giảm nhiều nhất có thể. Stack duy trì tăng dần = đảm bảo không còn cặp nào sai thứ tự (lớn trước nhỏ). Chứng minh: Giả sử PA tối ưu xóa d[j] (j > i) mà d[i] > d[i+1]. Đổi: xóa d[i] thay d[j]. Số kết quả nhỏ hơn (vì d[i] ở vị trí cao hơn và lớn hơn d[i+1]) → PA ban đầu không tối ưu.

VD: s = "1432", K = 1. Stack: push '1'. push '4' (4>1 OK). Gặp '3': đỉnh='4' > '3' và K=1 > 0 → pop '4' (xóa 4), K=0, push '3'. Push '2' (2<3 nhưng K=0 nên không xóa được). Stack = "132". VD khác: "10200" K=1 → gặp '0': đỉnh '1' > '0', pop '1', K=0. Kết quả: "0200" = "200". O(N).

📝 Code C++ — E14. Earliest Deadline First 
// Bài 14: TREHAN — Sắp xếp theo hạn chót (EDF)
#include <bits/stdc++.h>
using namespace std;

bool cmp(pair<int,int> a, pair<int,int> b) {
    return a.second < b.second;
}

int main() {
    ifstream fin("TREHAN.INP");
    ofstream fout("TREHAN.OUT");
    int n;
    fin >> n;
    vector<pair<int,int>> jobs(n); // (duration, deadline)
    for (int i = 0; i < n; i++) fin >> jobs[i].first >> jobs[i].second;
    sort(jobs.begin(), jobs.end(), cmp);
    long long time = 0, maxLate = 0;
    for (int i = 0; i < n; i++) {
        time += jobs[i].first;
        long long late = time - jobs[i].second;
        if (late > maxLate) maxLate = late;
    }
    fout << maxLate;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam — Earliest Deadline First (B2).

🎯 Tham lam thế nào? Sắp xếp công việc theo hạn chót tăng dần. Làm tuần tự từ đầu: việc hạn sớm làm trước. Biến time cộng dồn thời gian làm. Độ trễ việc i = time − deadline[i]. Đáp án = max(độ trễ).

🔎 Vì sao làm việc hạn sớm trước là tối ưu? (Đây là bài kinh điển EDF!) Hãy tưởng tượng bạn có nhiều bài tập, mỗi bài có hạn nộp. Bạn muốn độ trễ (độ trễ MAX) nhỏ nhất. Nếu bạn làm bài hạn XA trước bài hạn GẦN: bài hạn gần bị đẩy lùi → trễ nhiều (vì hạn gần mà làm muộn). Bài hạn xa được làm sớm → không bớt trễ bao nhiêu (vì hạn xa sẵn). Chứng minh hoán vị (exchange argument): Giả sử đổi 2 việc liên kề A (hạn d_A) và B (hạn d_B, d_B > d_A). Trước: A làm tại t, B làm tại t+p_A. Trễ A = t+p_A − d_A, trễ B = t+p_A+p_B − d_B. Sau khi đổi: B làm tại t, A làm tại t+p_B. Trễ B = t+p_B − d_B, trễ A = t+p_B+p_A − d_A. max(trễ A, trễ B) sau đổi: trễ A = t+p_B+p_A − d_A > t+p_A − d_A (trễ A trước). Vì d_A < d_B nên đổi làm trễ A tăng → max trễ tăng → đổi là TỆ HƠN.

VD: Việc: A(thời gian=2, hạn=4), B(3, hạn=10), C(1, hạn=6). Sắp theo hạn: A(4), C(6), B(10). Làm: A xong lúc 2, trễ 2−4=−2(OK). C xong lúc 3, trễ 3−6=−3(OK). B xong lúc 6, trễ 6−10=−4(OK). Max trễ = −2. Nếu làm B trước: B xong 3, trễ −7. A xong 5, trễ 5−4=1. C xong 6, trễ 0. Max = 1 — tệ hơn! O(N log N).

📝 Code C++ — E15. DSU Scheduling 
// Bài 15: LICHLAMVIEC — Sort by profit + DSU tìm slot trống
#include <bits/stdc++.h>
using namespace std;

int par[100001];

int find(int x) {
    while (par[x] != x) {
        par[x] = par[par[x]]; // path halving
        x = par[x];
    }
    return x;
}

bool cmp(pair<int,int> a, pair<int,int> b) {
    return a.second > b.second;
}

int main() {
    ifstream fin("LICHLAMVIEC.INP");
    ofstream fout("LICHLAMVIEC.OUT");
    int n;
    fin >> n;
    vector<pair<int,int>> jobs(n); // (deadline, profit)
    int maxD = 0;
    for (int i = 0; i < n; i++) {
        fin >> jobs[i].first >> jobs[i].second;
        maxD = max(maxD, jobs[i].first);
    }
    for (int i = 0; i <= maxD; i++) par[i] = i;
    sort(jobs.begin(), jobs.end(), cmp);
    long long ans = 0;
    for (int i = 0; i < n; i++) {
        int slot = find(jobs[i].first);
        if (slot > 0) {
            ans += jobs[i].second;
            par[slot] = slot - 1;
        }
    }
    fout << ans;
    return 0;
}
💡 Ý tưởng: Dùng Tham lam (B2) + DSU (D3).

🎯 Tham lam thế nào? Sắp việc theo lợi nhuận GIẢM. Duyệt từ việc lời nhiều nhất: với việc (lợi p, hạn d), tìm slot lớn nhất ≤ d còn trống. Nếu có → đặt việc vào slot đó, cộng p vào đáp án. Không có → bỏ việc (không kịp hạn nào).

🔎 Vì sao ưu tiên lợi nhuận cao? Vì sao làm càng muộn càng tốt?
(1) Lợi cao trước: Việc lợi 100đ luôn đáng làm hơn việc lợi 10đ. Nếu 2 việc tranh cùng slot, bỏ việc rẻ hơn là đúng.
(2) Làm càng muộn càng tốt (slot lớn nhất): Việc hạn ngày 5, làm ngày nào cũng được (1,2,3,4,5). Làm ngày 5 (để dành ngày 1-4 cho việc khác có hạn sớm hơn). Nếu làm ngày 1, việc khác hạn ngày 1 sẽ không có chỗ!

🔧 Vì sao dùng DSU? Tìm slot trống gần nhất ≤ d: cách thường là vòng lặp từ d xuống 1 → O(d). DSU: find(d) = slot trống gần nhất. Khi dùng slot s: union(s, s−1) → lần sau find(s) trả về s−1 (slot trống kế tiếp). Hình dung: các slot như dãy ghế. Khi ngồi ghế s, "nối" nó với ghế s−1. Lần sau ai tìm ghế ≤ s sẽ được chỉ thẳng đến s−1. O(α(N)) gần O(1) mỗi truy vấn.

VD: Việc: A(lợi 100, hạn 2), B(lợi 50, hạn 1), C(lợi 10, hạn 2). Sắp giảm: A, B, C. Slot [1, 2] trống. A(hạn 2): find(2)=2, đặt A → slot 2, union(2,1). B(hạn 1): find(1)=1, đặt B → slot 1, union(1,0). C(hạn 2): find(2)→find(1)→find(0)=0 < 1 → không có slot → bỏ C. Tổng = 100+50 = 150. O(N log N).
